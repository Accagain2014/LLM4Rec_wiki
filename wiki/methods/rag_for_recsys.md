# 面向推荐系统的检索增强生成

## 摘要

面向推荐系统的检索增强生成（RAG）将**外部信息检索**与**LLM 生成**相结合，以产生更准确、更有根据、信息更丰富的推荐输出。RAG 不单纯依赖 LLM 的参数化知识（可能过时或不正确），而是在生成推荐之前检索相关的**用户评论**、**项目描述**、**知识图谱路径**和**上下文信息**。

## 要点

- RAG 将 LLM 输出锚定在**检索证据**上，减少幻觉
- 两阶段流水线：**检索**相关文档 → **生成**推荐
- 检索来源：用户评论、项目元数据、知识图谱、相似用户画像
- 提升推荐的**忠实性**和**可验证性**
- 增加计算开销但提升了质量

## 详情

### RAG 用于推荐的架构

```
用户查询/上下文
       │
       ▼
┌─────────────┐
│   检索器    │  ← 从以下检索：项目数据库、评论、知识图谱、用户历史
└──────┬──────┘
       │ 检索到的文档
       ▼
┌─────────────┐     ┌──────────────┐
│   提示      │ ──→ │  LLM (Qwen)  │
│   构建器    │     │   生成器     │
└─────────────┘     └──────┬───────┘
                           │
                           ▼
                  推荐 + 解释
```

### 检索来源

| 来源 | 提供内容 | 示例 |
|--------|------------------|---------|
| **项目评论** | 真实用户观点 | 候选项目的前 10 条评论 |
| **项目元数据** | 事实属性 | 类型、导演、演员、上映年份 |
| **知识图谱** | 关系结构 | "由克里斯托弗·诺兰执导"路径 |
| **相似用户** | 协同信号 | 品味相似的用户画像 |
| **用户历史** | 个人上下文 | 用户过往交互和评论 |
| **外部知识** | 世界知识 | 维基百科文章、专业评论 |

### 检索策略

**稠密检索（Dense Retrieval）：**
- 将查询和文档编码为嵌入向量
- 使用向量相似度（余弦、内积）
- 工具：FAISS、Milvus、Elasticsearch（稠密向量）

**稀疏检索（Sparse Retrieval）：**
- 对文本使用 BM25、TF-IDF
- 适用于关键词匹配
- 工具：Elasticsearch、Lucene

**混合检索（Hybrid Retrieval）：**
- 结合稠密 + 稀疏检索，带学习权重
- 兼顾两者优势
- LLM 可辅助对结果加权

### RAG 提示构建

```
相关信息：
[检索到的评论 1]："这部电影视觉效果出色但剧情薄弱。"
[检索到的评论 2]："摄影效果堪称奥斯卡级别。"
[项目元数据]：导演：丹尼斯·维伦纽瓦 | 类型：科幻 | 时长：163分钟

用户上下文：
历史：[《银翼杀手2049》, 《降临》, 《沙丘》]
偏好："我喜欢视觉效果出色的科幻片"

任务：
基于以上信息，为该用户推荐《沙丘2》。
提供基于检索信息的具体理由。
```

### 对推荐系统的好处

1. **减少幻觉**：基于检索到的事实，而非捏造
2. **信息时效性**：检索反映当前项目属性
3. **可验证性**：用户可以查看来源评论/信息
4. **个性化**：检索到的相似用户画像增加了协同信号
5. **可解释性**：可以引用具体评论或事实作为依据

### 挑战

1. **检索质量**：垃圾进 → 垃圾出；检索必须相关
2. **上下文窗口**：检索文档消耗 Token 预算
3. **延迟**：两阶段流水线（检索 + 生成）更慢
4. **索引维护**：检索索引必须随新项目保持更新
5. **无关检索**：检索到的文档可能对推荐无帮助

### 高级 RAG 技术

**迭代式 RAG（Iterative RAG）：**
- 生成 → 识别知识缺口 → 再次检索 → 优化
- "我需要了解演员阵容才能给出好的推荐" → 检索演员表 → 更新推荐

**重排序（Reranking）：**
- 检索 100 个文档 → 用 LLM 重排序 → 使用前 10 个进行生成
- 提升检索上下文的相关性

**智能体式 RAG（Agentic RAG）：**
- LLM 自行决定检索*什么*以及*何时*检索
- "先让我查看评论，再与用户偏好对比"
