# OneRec-V2 — 惰性纯解码器生成式推荐

## 概述

OneRec-V2 是快手团队的第二代端到端生成式推荐模型，解决了 OneRec-V1 的两个关键瓶颈：（1）**计算效率低下**，97.66% 的资源消耗在序列编码而非生成上；（2）**强化学习受限**，仅依赖奖励模型。OneRec-V2 引入了**惰性纯解码器架构**，消除了编码器瓶颈，将总计算量减少 94%，训练资源减少 90%，使扩展到 8B 参数成为可能。它还结合了**感知时长的奖励塑造**和**自适应比率裁剪**，用于与真实用户交互进行偏好对齐。

## 要点

- **惰性纯解码器架构**：移除编码器，计算量减少 94%
- **扩展到 8B 参数**：由计算效率提升使能
- **感知时长的奖励塑造**：将观看时长信号纳入偏好对齐
- **自适应比率裁剪**：用真实反馈稳定 RL 训练
- **工业级部署**：快手 A/B 测试显示 App 停留时间 +0.467%/+0.741%
- **多目标推荐**：同时平衡多个参与度指标

## 详情

### 动机：V1 的局限性

OneRec-V1 虽然在生产中取得成功，但面临两个根本性的可扩展性挑战：

1. **编码器瓶颈**：97.66% 的计算资源被序列编码阶段消耗，留给实际生成任务的容量极小。这种编码器-解码器分离对于推荐来说效率极低，因为编码阶段占主导地位，而生成阶段才是核心价值所在。
2. **奖励模型依赖**：强化学习流水线仅依赖学习到的奖励模型，这会引入近似误差并限制与真实用户偏好对齐的质量。

### 惰性纯解码器架构

OneRec-V2 的关键架构创新是从编码器-解码器过渡到**纯解码器**设计：

```
User behavior tokens → Decoder (autoregressive) → Item generation
```

- **无单独编码器**：用户历史直接在解码器的自注意力中处理，消除了冗余的编码阶段
- **总计算量减少 94%**：通过移除编码器，整体 FLOPs 大幅下降
- **训练资源减少 90%**：每训练步骤所需内存和计算更少
- **扩展到 8B 参数**：效率提升使在相同基础设施预算内训练更大规模模型成为可能

这种"惰性"设计认识到：在推荐中，用户行为序列已经是离散 token（物品 ID），因此单独的连续编码步骤是不必要的开销。

### 与真实交互的偏好对齐

OneRec-V2 通过两项关键创新改进了 V1 的强化学习方法：

#### 感知时长的奖励塑造
- 将**实际观看/停留时长**作为奖励函数中的信号
- 超越二元点击/交互反馈，捕捉参与深度
- 塑造奖励以反映用户不仅*是否*交互，而且*交互了多久*
- 使模型优化与业务指标（停留时间、参与度）更好地对齐

#### 自适应比率裁剪
- 解决带噪声真实反馈的 RL 训练不稳定性
- 根据可用反馈信号的质量和数据量动态调整裁剪比率
- 防止对稀疏或有偏交互数据的过拟合
- 在正/负样本比例波动时稳定训练

### 多目标推荐

OneRec-V2 支持多目标优化，平衡：
- 点击率（CTR）
- 观看/停留时间
- 用户参与多样性
- 内容探索与利用

感知时长的奖励塑造自然地将多目标整合为统一的奖励信号。

### 性能

| 指标 | 结果 |
|------|------|
| App 停留时间提升 | +0.467% / +0.741%（A/B 测试场景） |
| 计算量减少 | 相比 OneRec-V1 减少 94% |
| 训练资源减少 | 相比 OneRec-V1 减少 90% |
| 模型规模 | 最高 8B 参数 |
| 部署 | 快手生产环境 |

### 与 OneRec-V1 对比

| 方面 | OneRec-V1 | OneRec-V2 |
|------|-----------|-----------|
| 架构 | 编码器-解码器 | 纯解码器（惰性） |
| 编码开销 | 占计算的 97.66% | 已消除 |
| 最大参数量 | 受计算限制 | 8B |
| RL 方法 | 仅奖励模型 | 感知时长 + 自适应裁剪 |
| A/B 测试指标 | +1.6% 观看时长 | +0.467%/+0.741% App 停留时间 |
